## 二、数据库方案：SQLite（GitHub 友好）

### 为什么选 SQLite？

| 对比项 | 原方案 MySQL | 新方案 SQLite |
|--------|-------------|---------------|
| 部署复杂度 | 需安装 MySQL 服务 | 零配置，单文件 |
| 环境依赖 | 需 mysql2 驱动 | 仅需 better-sqlite3 |
| 备份/迁移 | 导出 SQL 脚本 | 直接复制 .db 文件 |
| GitHub 兼容性 | 不适合存 GitHub | 可随代码仓库管理 |
| 适合场景 | 高并发大型项目 | 个人站、数据量小 |

**结论**：米米的小世界是个人展示站，数据量有限，SQLite 完全满足需求，且极大简化部署和备份。

### 数据库文件位置

```
项目根目录/
├── data/
│   └── app.db          ← SQLite 数据库文件（可 Git 备份）
├── db/
│   ├── schema.ts       ← 表结构定义（已迁移为 SQLite 语法）
│   └── migrations/     ← Drizzle 迁移文件
```

### 关键迁移变化

| 原 MySQL 类型 | 新 SQLite 类型 | 说明 |
|---------------|---------------|------|
| `serial` | `integer` + `autoIncrement` | 自增主键 |
| `varchar(n)` | `text` | 变长字符串 |
| `timestamp` | `integer` + `mode: "timestamp"` | 时间戳 |
| `mysqlEnum` | `text` + `default` | 枚举值 |
| `onDuplicateKeyUpdate` | `onConflictDoUpdate` | 冲突处理 |

## 三、安全修复（高危 → 已修复）

### ✅ 3.1 密码使用 bcrypt 哈希（原：Base64 编码）

**文件**：`api/routers/localAuthRouter.ts`

**原代码**：
```typescript
const passwordHash = Buffer.from(input.password).toString("base64");
```

**修复后**：
```typescript
import bcrypt from "bcryptjs";
const passwordHash = await bcrypt.hash(input.password, 10);
const valid = await bcrypt.compare(input.password, user.passwordHash);
```

**影响**：密码不可逆存储，即使数据库泄露也无法获取原始密码。

---

### ✅ 3.2 SQL 注入风险修复（原：字符串拼接 IN 子句）

**文件**：`api/routers/likeRouter.ts`

**原代码**：
```typescript
where(sql`${likes.imageId} IN (${input.join(",")})`)
```

**修复后**：
```typescript
import { inArray } from "drizzle-orm";
where(inArray(likes.imageId, input))
```

**影响**：彻底消除 SQL 注入风险，参数由 Drizzle ORM 自动转义。

---

### ✅ 3.3 本地认证 Token 使用 JWT 签名（原：Base64 JSON）

**文件**：`api/context.ts`、`api/routers/localAuthRouter.ts`、`src/hooks/useAuth.ts`

**原代码**：
```typescript
const token = Buffer.from(JSON.stringify({ id, name, role })).toString("base64");
```

**修复后**：
```typescript
import { SignJWT, jwtVerify } from "jose";
const token = await new SignJWT({ id, name, role })
  .setProtectedHeader({ alg: "HS256" })
  .setIssuedAt()
  .setExpirationTime("7d")
  .sign(JWT_SECRET);
```

**影响**：Token 带签名和过期时间，无法被篡改伪造管理员身份。

---

### ✅ 3.4 文件上传增加内容校验（原：仅检查 MIME）

**文件**：`api/lib/upload.ts`

**新增**：
- 文件大小限制（10MB）
- MIME 类型白名单（JPEG/PNG/WebP/GIF）
- 魔数（Magic Bytes）校验文件内容真伪
- 随机安全文件名（避免原始文件名暴露信息）

---

### ✅ 3.5 添加接口速率限制

**文件**：`api/boot.ts`、`api/lib/rateLimiter.ts`

**新增**：
| 接口 | 限制 |
|------|------|
| 登录 | 每分钟 5 次 |
| 注册 | 每分钟 3 次 |
| 文件上传 | 每分钟 10 次 |

## 七、GitHub 部署指南

### 步骤 1：初始化 Git 仓库

```bash
cd 项目目录

git init
git add .
git commit -m "feat: 优化上线 — 安全加固 + SQLite 迁移 + 功能修复"
```

### 步骤 2：创建 GitHub 仓库

1. 访问 https://github.com/new
2. 创建新仓库（如 `mimi-cat-showcase`）
3. 按页面提示推送代码：

```bash
git remote add origin https://github.com/你的用户名/mimi-cat-showcase.git
git branch -M main
git push -u origin main
```

### 步骤 3：数据库备份策略

SQLite 数据库文件 `data/app.db` 可以加入 Git 进行版本管理：

```bash
# 确保 data 目录在 git 中
git add data/app.db
git commit -m "chore: 备份数据库"
```

**建议**：定期提交数据库备份，或在 `.github/workflows` 中配置自动备份 Action。

### 步骤 4：服务器部署

#### 方案 A：自有服务器 / VPS（推荐）

```bash
# 克隆代码
git clone https://github.com/你的用户名/mimi-cat-showcase.git
cd mimi-cat-showcase

# 安装依赖
npm install

# 创建环境变量文件
cp .env.example .env
# 编辑 .env，填入 APP_ID、APP_SECRET、KIMI_AUTH_URL 等

# 运行数据库迁移
npm run db:migrate

# 构建前端 + 后端
npm run build

# 启动生产服务
npm start
```

#### 方案 B：Docker 部署（可选）

如需 Docker 支持，可添加 Dockerfile：

```dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build
EXPOSE 3000
CMD ["npm", "start"]
```
